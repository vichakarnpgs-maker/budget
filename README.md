# ระบบบริหารแผนปฏิบัติการประจำปีงบประมาณ — โรงเรียนโพนงามศึกษา

ระบบเว็บแอปสำหรับบันทึก ติดตาม และสรุปผลการใช้จ่ายงบประมาณตามแผนปฏิบัติการ
แบ่งเป็น 5 กลุ่มงาน พร้อมระบบสิทธิ์การเข้าถึงรายบุคคล (Google Sign-In)

## โครงสร้างไฟล์

| ไฟล์ | หน้าที่ |
|---|---|
| `index.html` | หน้าเข้าสู่ระบบ (Google Sign-In) |
| `dashboard.html` | แดชบอร์ดภาพรวมโรงเรียน + เปรียบเทียบ 5 กลุ่มงาน |
| `data-entry.html` | หน้าบันทึก/แก้ไขข้อมูลงบประมาณ รายกลุ่มงาน |
| `import.html` | หน้านำเข้าข้อมูลจาก CSV/Excel |
| `shared.js` | โครงสร้างข้อมูล, การคำนวณ roll-up, ตัวแปลง CSV, การเชื่อมต่อ backend |
| `auth.js` | การเข้าสู่ระบบและตรวจสอบสิทธิ์ฝั่ง client |
| `theme.css` / `theme.js` | ธีมสว่าง/มืด |
| `icons.js` | ชุดไอคอน SVG (ไม่ใช้อีโมจิ) |
| `apps-script/Code.gs` | โค้ด backend (Google Apps Script) — **ส่วนความปลอดภัยตัวจริง** |
| `academic-sample.csv` | ตัวอย่างไฟล์นำเข้า แปลงจากไฟล์ Excel ต้นแบบกลุ่มวิชาการที่แนบมา ใช้ทดสอบหน้า "นำเข้าข้อมูล" ได้ทันที |

## หลักการความปลอดภัย (สำคัญ — อ่านก่อนใช้งานจริง)

ระบบนี้ใช้สถาปัตยกรรม **"client แสดงผล + server ตัดสินใจ"**:

1. ผู้ใช้ login ด้วยบัญชี Google ผ่าน Google Identity Services (`index.html`)
2. ทุกครั้งที่มีการ **โหลด/บันทึก** ข้อมูล หน้าเว็บจะส่ง Google ID Token ไปให้
   `apps-script/Code.gs` ซึ่งจะ:
   - ตรวจสอบลายเซ็น Token กับ Google โดยตรง (ป้องกันการปลอมอีเมล)
   - ตรวจสอบอีเมลกับชีต `Whitelist` ว่ามีสิทธิ์หรือไม่ และมีสิทธิ์แก้ไขรายการ/กลุ่มงานใดบ้าง
   - หากเป็นการบันทึกข้อมูล (`saveData`) เซิร์ฟเวอร์จะ **merge ตามสิทธิ์จริง**
     กล่าวคือ แม้ client จะส่งข้อมูลทั้งชุดมา เซิร์ฟเวอร์จะรับเฉพาะรายการที่
     ผู้ใช้คนนั้นมีสิทธิ์แก้ เท่านั้น รายการอื่นจะคงค่าฝั่งเซิร์ฟเวอร์ไว้เสมอ
     แม้ client จะถูกดัดแปลง state ก็ไม่สามารถเขียนทับข้อมูลของผู้อื่นได้
3. การล็อก/ซ่อนปุ่มฝั่งหน้าเว็บ (`auth.js`, `data-entry.html`) เป็นเพียงการ
   ปรับปรุงประสบการณ์ใช้งาน (UX) เท่านั้น **ไม่ใช่ตัวป้องกันจริง**

> **ข้อควรระวัง**: ถ้ายังไม่ได้ตั้งค่า `APPS_SCRIPT_URL` ระบบจะทำงานในโหมดทดสอบ
> (เก็บข้อมูลใน localStorage ของเครื่อง ไม่มีการตรวจสอบสิทธิ์จริง) ใช้สำหรับ
> ทดลองหน้าจอเท่านั้น **ห้ามใช้งานจริงก่อนตั้งค่า backend**

## อัปเดตล่าสุด (รอบที่ 3)

- **แก้บั๊ก**: โครงการ/งานที่ไม่มีกิจกรรมย่อยเลย (ลูกเป็นศูนย์) แต่เดิมกรอกยอดใช้จ่ายไม่ได้
  เพราะระบบเช็คเฉพาะ type ≠ "project" ตอนนี้แก้เป็น "ไม่มีลูกเลย = leaf" กรอกได้ทุกระดับ
- เปลี่ยนจำนวนครั้งการเบิกจ่ายเริ่มต้นจาก 7 เป็น **8 ครั้ง**
- เปลี่ยนการบันทึกยอดใช้จ่ายจาก "คลิกแก้ตรงเซลล์" เป็น **ป๊อปอัพเดียวจบ**: กรอกจำนวนเงิน
  + วันที่เบิกจ่าย + คะแนนความพึงพอใจ (0-5) ต่อครั้ง พร้อมแก้งบแยกประเภทและเป้าหมายเชิงปริมาณในป๊อปอัพเดียวกัน
- คอลัมน์ "การเบิกจ่ายรายครั้ง" ในตารางแสดงเฉพาะไอคอน ✓ เมื่อมีการบันทึกแล้ว (ลดความกว้างคอลัมน์)
  รายละเอียดเต็ม (จำนวนเงิน/วันที่) ดูได้จาก tooltip หรือเปิดป๊อปอัพ
- เพิ่มสูตรแปลผลอัตโนมัติจากคะแนนความพึงพอใจเฉลี่ย (เทียบบัญญัติไตรยางศ์เป็นร้อยละ 100):
  **เชิงคุณภาพ** (ดีเยี่ยม/ดีมาก/ดี/พอใช้/ปรับปรุง) และ **ผลสรุปโครงการ** (สูงกว่า/เป็นไปตาม/ใกล้เคียง/
  ต่ำกว่าค่าเป้าหมาย/ต้องปรับปรุงเร่งด่วน) ที่เกณฑ์ ≥87.75 / ≥62.75 / ≥37.75 / ≥12.75 / ต่ำกว่า
- สถานะ "แจ้งเตือนการใช้จ่าย" เปลี่ยนข้อความเป็น "ใช้จ่ายเกินงบประมาณ" / "สมดุลงบประมาณ" ตามสูตรที่กำหนด
- แดชบอร์ดเพิ่มตัวชี้วัดนอกเหนือจากงบประมาณ: จำนวนโครงการรวม, ผลสรุปภาพรวมโรงเรียน,
  จำนวนรายการใช้จ่ายเกินงบ, จำนวนรายการรอเบิกจ่าย ทั้งระดับโรงเรียนและรายกลุ่มงาน
- เพิ่มโหมด **DEV_MODE** ใน `auth.js` (ค่าเริ่มต้นเปิดอยู่) ข้ามการ login จริงเพื่อดู UX/UI ได้ทันที
  ปิดโดยตั้ง `DEV_MODE = false` เมื่อพร้อมใช้งานจริงพร้อม Google OAuth

## ขั้นตอนติดตั้งใช้งานจริง

### 1) สร้าง Google OAuth Client ID
- ไปที่ [Google Cloud Console](https://console.cloud.google.com/) > APIs & Services > Credentials
- สร้าง OAuth Client ID ประเภท **Web application**
- ใส่โดเมนที่จะใช้ host เว็บนี้ใน Authorized JavaScript origins
- คัดลอก Client ID ไปวางที่ตัวแปร `GOOGLE_CLIENT_ID` ใน `auth.js`

### 2) ตั้งค่า Backend (Google Apps Script)
- สร้าง Google Sheet ใหม่ 1 ไฟล์ → Extensions > Apps Script → วางโค้ดจาก `apps-script/Code.gs`
- ไปที่ Project Settings > Script Properties เพิ่ม `GOOGLE_CLIENT_ID` ให้ตรงกับขั้นตอน 1)
- สร้างชีตชื่อ `Whitelist` พร้อมหัวคอลัมน์ (ดูตัวอย่างละเอียดในคอมเมนต์บนสุดของ `Code.gs`):

  | Email | Role | Departments | EditableActivityIDs |
  |---|---|---|---|
  | director@school.ac.th | admin | | |
  | teacher.a@school.ac.th | editor | academic | |

- Deploy > New deployment > Web app (Execute as: Me, Who has access: Anyone)
- คัดลอก URL ที่ได้ไปใส่ในตัวแปร `APPS_SCRIPT_URL` ของ `shared.js`

### 3) อัปโหลดไฟล์เว็บ
อัปโหลดไฟล์ทั้งหมด (ยกเว้นโฟลเดอร์ `apps-script/`) ขึ้น hosting ใดก็ได้ที่รองรับ
static site (เช่น Google Sites แบบฝัง iframe, GitHub Pages, หรือพื้นที่เว็บของโรงเรียน)

### 4) ทดสอบนำเข้าข้อมูล
เปิดหน้า `import.html` แล้วอัปโหลด `academic-sample.csv` (แปลงจากไฟล์ Excel ต้นแบบ
ที่แนบมา) เพื่อทดสอบว่าระบบจัดลำดับชั้นโครงการ/งาน/กิจกรรมถูกต้อง

## การปรับแต่งกลุ่มงาน 5 กลุ่ม

แก้ชื่อ/สี/รหัสกลุ่มงานได้ที่ตัวแปร `DEPARTMENTS` ใน `shared.js` จุดเดียว
ระบบทั้งหมด (เมนู, แดชบอร์ด, หน้าบันทึก, หน้านำเข้า) จะอัปเดตตามทันที
ปัจจุบันตั้งกลุ่มที่ 5 เป็น **"กลุ่มบริหารงานบุคคล"** ไว้ชั่วคราว (ตามโครงสร้าง
4+1 กลุ่มงานมาตรฐานของสถานศึกษา) หากชื่อจริงของโรงเรียนแตกต่าง แก้ได้ที่บรรทัด
`{ id: 'personnel', name: '...', short: '...', color: '...' }`

## กฎการนำเข้า CSV

อ้างอิงจากโครงสร้างไฟล์ Excel ต้นแบบที่แนบมา คอลัมน์ตามลำดับ:

```
ที่ | ชื่อโครงการ/งาน/กิจกรรม | งบจัดสรร | อุดหนุน | พัฒนาผู้เรียน | รายได้ฯ | เชิงปริมาณ | เชิงคุณภาพ | ผู้รับผิดชอบ | หมายเหตุ | (อีเมลผู้รับผิดชอบ — เพิ่มเองได้)
```

- **เลขเต็ม** (1, 2, 3...) → ระดับ "โครงการ"
- **เลขทศนิยม X.Y** (1.1, 8.2...) → ระดับ "งาน" (ลูกของโครงการล่าสุด)
- **ว่าง** → ถ้ามี "งาน" อยู่ก่อนหน้า เป็น "กิจกรรมย่อย" ของงานนั้น
  ถ้าไม่มี เป็น "งาน" แบบไม่มีเลข (ลูกตรงของโครงการ)
- รายการใดไม่มีลูกต่อ (leaf) ไม่ว่าจะเป็นระดับงานหรือกิจกรรม จะกรอกยอดใช้จ่าย
  รายครั้งได้โดยตรงในหน้าบันทึกข้อมูล (คลิกที่ช่องตัวเลขแล้วพิมพ์ได้ทันที)

## ฟีเจอร์เด่นของเวอร์ชันนี้

- แยกหน้าแดชบอร์ดออกจากหน้าบันทึกข้อมูล พร้อมกราฟ/ตารางเปรียบเทียบ 5 กลุ่มงาน
- คลิกกรอกยอดใช้จ่ายรายครั้งได้ตรงในตาราง ไม่ต้องเปิด modal
- แถวสรุปท้ายตาราง (footer) รวมทุกคอลัมน์ตัวเลขให้สอดคล้องกับหัวตาราง
- โหมดมืด/สว่าง สลับได้ทุกหน้า จำค่าไว้อัตโนมัติ
- ไอคอนทั้งหมดเป็น SVG แบบมืออาชีพ ไม่ใช้อีโมจิ
- ระบบสิทธิ์รายบุคคล (whitelist) ตรวจสอบจริงฝั่งเซิร์ฟเวอร์ทุกครั้งที่บันทึก
- หน้านำเข้า CSV พร้อมตัวอย่างข้อมูลก่อนยืนยัน และเลือกโหมด "แทนที่" หรือ "เพิ่มต่อท้าย"
